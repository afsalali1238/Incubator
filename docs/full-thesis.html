№02 / 10 THE PROBLEM
Most founders don't fail at building.
They fail at deciding what to build.

AI coding tools make building feel free. Claude can ship an MVP in an afternoon. So the thinking phase gets skipped entirely — because it doesn't feel like work, and the building does.

WHAT GETS SKIPPED
  • Who specifically has this problem
  • Why this moment — not "AI makes it possible"
  • What the competitive graveyard looks like
  • Which assumption, if wrong, kills everything
  • What the de-risking sequence is
  • Who to hire first and why
WHAT ACTUALLY MATTERS
  • A named user in a specific painful situation
  • A structural change that made now different
  • Cause-of-death patterns from dead analogues
  • The single assumption specific to this project
  • The order winners followed — not features, sequence
  • Roles justified by research findings, not templates
"Beautiful products built for imaginary users — that's not a building problem. That's a thinking problem."

№03 / 10 THE INSIGHT
Not a prompt.
A protocol.

The thinking a good investor does before writing a check — and the thinking a good advisor does before designing a team — can be systematized. That's the core insight.

A prompt is a suggestion. A protocol has state, phases, checkpoints, and conditions under which it pushes back and refuses to proceed. Project CEO is a protocol system: 7 phases, an explicit state machine, and a CEO that can tell you "I'm not starting research on this thesis yet."

PROTOCOL HAS STATE session-state.json written after each phase. Authoritative resume anchor. The CEO finds your project automatically next session.
PROTOCOL CHECKS DRIFT DRIFT CHECK before synthesis — compares research signals against Phase 1 thesis on 4 dimensions. ASSUMPTION AUDIT before every board meeting.
PROTOCOL PUSHES BACK Three hardcoded triggers for vague answers. A HOLD condition at thesis lock. A Heresy panel that can say "Don't build this."
PROTOCOL HAS MODES LOW_SIGNAL, CONFLICT, PARTIAL_RESUME — explicit degradation modes with defined behavior for when things go wrong mid-session.

№05 / 10 THE PROTOCOL LAYER
State management
for a stateless model.

The hardest engineering problem in prompt systems isn't intelligence — it's continuity. A language model has no persistent memory between sessions. Project CEO solves this with an explicit file-based state machine.

session-state.json Written after Phase 2, 3, and 4. Contains phase, status, searches_used, drift_flags, key_assumptions, confidence, open_risks, unresolved_assumptions, and a 20-word summary. Read first on every resume. File-existence inference is fallback only.
STATE_JSON in roster.md Canonical source of truth for agent team state. CEO reads and writes JSON block first, regenerates markdown table from it. Markdown is presentation only. The two cannot diverge.
DRIFT CHECK Runs at Phase 2 synthesis trigger. Checks 4 dimensions: scope, audience, model, stage. 2+ flags → "⚠️ DRIFT DETECTED" before synthesis. Injected into Panel 01 of the report.
ASSUMPTION AUDIT Runs before every board meeting. Re-reads Phase 1 thesis against current agent outputs. Any contradiction surfaces as "⚠️ ASSUMPTION DRIFT" before the agenda starts.
Confidence labels 🔵 Multiple independent sources. 🟡 Partial or conflicting. 🔴 Inferred, single source, or INSUFFICIENT DATA. Applied inline to every key finding in the report.

№06 / 10 THE TEAM ENGINE
The CEO never does
domain work directly.

Every role on the roster is hired because a specific research finding demanded it — from the Graveyard (what killed analogues), the Winners (what they built first), the Heresy (what the kill shot is). No finding, no hire.

HIRE Name precisely → persona brief → skill file → first task immediately. A new hire with no task is an organizational smell. Every role gets their first delegation the day they're created.
FIRE Two consecutive 🔴 board meetings = fire, not monitor. "That's a fire, not a flag. Here's who takes the open work." No softening to "reduced scope" or "strategic realignment."
SPAWN One-off domain expertise that doesn't warrant a standing hire: IP lawyer, unit economics model, naming session. Task closes, temp agent is archived, outcome logged to roster.
DELEGATE Every delegation has: a measurable task, an exit criterion, a scope fence, and curated context. CEO reviews on correctness, completeness, and fit. Accept / Return with feedback / Re-delegate / Escalate.
QUALITY GATE [CRITIQUE GATE] prompt required after every skill. Only two valid outputs: "Quality gate: PASS" or "Quality gate: FAIL — rewriting [section]." No other path forward.

Health scores: 🟢 delivered + reviewed · 🟡 partial or blocked · 🔴 nothing delivered · 🟡(p) provisional for unreviewed work

№07 / 10 THE HONESTY LAYER
The CEO pushes back.
Every time. By design.

Most AI tools in the "business advisor" category are optimized to make the user feel good. Project CEO is optimized to make the user think clearly. Three hardcoded pushback triggers fire before the CEO moves to the next question:

BROAD SEGMENT ANSWER: "Who has this problem?" → "small businesses" / "developers" / "parents"
"That's a segment, not a person. Give me a specific job title, situation, and what they were doing when they felt this. I'll wait."
GENERIC TIMING: "Why now?" → "the market is growing" / "AI makes it possible"
"That's true for 50 products launching this month. What changed in the last 12 months that makes this the right moment — specifically for you?"
UNIVERSAL ASSUMPTION: "Riskiest assumption?" → "people will use it" / "they'll pay for it"
"That's every startup's riskiest assumption. What assumption is specific to this project — the one only you are making, that if wrong, makes this particular thing pointless?"
"Proceed with caution" — when the evidence says don't build — is just flattery with better vocabulary. The Heresy panel can say "Don't build this. Here's why:" and mean it.

Pivot ≠ Refine. Pivot means the thesis is wrong. Refine means execution was imprecise. They are not the same call.

№09 / 10 THE BOUNDARIES
What it genuinely
cannot do.

Every serious protocol system should be honest about its limits. These are not design failures — they are the fundamental boundary between promptware and real orchestration. No amount of SKILL.md refinement crosses them.

NO RUNTIME SCHEMA ENFORCEMENT
There's no executor. Instructions can be followed imperfectly under context pressure. The model reconstructs state from disk — it doesn't maintain it in memory.
NO TRUE ASYNC MULTI-AGENT
Agents are serialized through one inference stream. There's no parallel execution, no independent agent memory, no true disagreement between agents.
NO DETERMINISTIC ORCHESTRATION
Long context accumulates entropy. Under pressure, the model may drift from instructions. This is a substrate limit — not a skill design problem.
NO OBSERVABLE KPIs
The CEO cannot watch live metrics, read analytics dashboards, or observe what's actually happening in the product. It knows what you tell it.
NO ENTERPRISE AUDITABILITY
No formal audit trail, no access control, no separation of concerns at the infrastructure level. This needs external tooling — it cannot live in a skill file.
NOT A REAL COMPANY
The most honest limit: this is structured strategic decomposition — not simulated company consciousness. The more "executive theater" language used, the more sophisticated users will notice.

CURRENT RATING — INDEPENDENT REVIEW

Project CEO is for solo founders, indie builders, and non-technical entrepreneurs who want the thinking done before the building starts. It is not a replacement for real advisors, a legal/compliance tool, or an enterprise orchestration platform.

WHAT IT DOES WELL
  • Forces thesis specificity before research
  • Autonomous market research with hard caps
  • Graveyard analysis — cause-of-death patterns
  • Roster design from research, not templates
  • Resumable state across sessions
  • Pushes back when the evidence demands it
WHAT IT DOESN'T DO
  • Real-time market data or live APIs
  • True parallel agent execution
  • Replace a domain expert or advisor
  • Guarantee research accuracy
  • Enforce its own schemas at runtime
  • Build the product
"It makes you think before you build. That's the whole product."
